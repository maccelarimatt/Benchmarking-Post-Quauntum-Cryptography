Origins & history

MAYO is a post-quantum, multivariate-quadratic (MQ) signature scheme that builds on Oil-and-Vinegar (OV) ideas. It was proposed by Beullens as a practical OV-based design with small public keys [1]. MAYO entered NIST’s Additional Digital Signatures track in 2023 and advanced to Round 2 in Oct. 2024 [2], with broader process details in NISTIR 8528 [3]. Round-2 specification updates were published in 2025 (see also the Round-1 spec) [4], [5].

Design overview

An OV public key is a quadratic map \(P:\mathbb{F}_q^n\!\to\!\mathbb{F}_q^m\) with a hidden oil subspace \(O\subset\mathbb{F}_q^n\) of dimension \(o\) such that \(P(o)=0\) for all \(o\in O\). Classic OV signs by fixing “vinegar” variables so the remaining equations are linear in the “oil” variables [6]. MAYO deliberately chooses a small \(o\) (shrinking the public key) which makes the system underdetermined; it then “whips” multiple inputs together to regain solvability while keeping the public key compact [1], [4].

Mathematical setting

Let \(P=(p_1,\dots,p_m)\) be quadratic polynomials over \(\mathbb{F}_q\) in \(n\) variables. Write the (symmetric) bilinear differential

\(P'(x,y)=P(x+y)-P(x)-P(y).\)

For a secret oil subspace \(O\subset\mathbb{F}_q^n\) of dimension \(o\), OV ensures \(P(o)=0\) for all \(o\in O\) [6]. MAYO introduces a public “whipped” map on \(k\) blocks [1], [4]:

\( P^\ast(x_1,\dots,x_k)=\sum_{i=1}^{k}E_{ii}P(x_i)+\sum_{1\le i < j \le k}E_{ij}P'(x_i,x_j) \).

with fixed public matrices \(E_{ij}\in\mathbb{F}_q^{m\times m}\). The zero set contains \( O_k=\{(o_1,\dots,o_k):o_i\in O\} \), giving an effective “oil space” of dimension \(k\,o\). Choosing parameters with \(k\,o > m\) lets signing proceed by solving a linear system in the oil coordinates [1].

Digest mapping & signature length

A message \(M\) is hashed (and salted) into a target \(t\in\mathbb{F}_q^m\). For \(q=16\), signatures encode \(kn\) field elements at 4 bits each, so the core signature part costs \(\lceil kn/2\rceil\) bytes plus a salt [4], [5].

Algorithms

Key generation

From seeds, derive a description of the oil space \(O\) and the public components of \(P\). The public key encodes a compact seed (and small “oil-block” data); expanded keys reconstruct full polynomial blocks deterministically [4], [5].

Signing

  1. Compute \(t\leftarrow \mathrm{H}(M,\text{salt})\in\mathbb{F}_q^m\).
  2. Sample \(k\) vinegar vectors \(v_i\in\mathbb{F}_q^{n-o}\) and build a linear system \(A x = y\) in the oil coordinates \(x\in\mathbb{F}_q^{k o}\) using \(P^\ast\), \(t\), and bilinear terms (e.g., from \(P'(v_i,v_j)\)) [1], [4].
  3. If solvable, recover \(o_i\) from \(x\), set \(s_i=v_i+o_i\), and output \(\sigma=(s_1,\dots,s_k,\text{salt})\). Otherwise, resample (low restart probability by design) [4].

Verification

  1. Recompute \(t\leftarrow \mathrm{H}(M,\text{salt})\).
  2. Evaluate \(P^\ast(s_1,\dots,s_k)\) with the public key and accept iff it equals \(t\) [4].

Security at a glance

  • Assumptions: hardness of recovering \(O\) (small-oil OV) and preimage resistance of the whipped map \(P^\ast\); EUF-CMA argued in the ROM [1], [4].
  • Rejection sampling: signing may restart with small probability; parameters bound leakage (negligible loss factors) [4].
  • Attacks considered: classical OV key-recovery lines (UOV, etc.) [6], improved solving for underdetermined MQ [8], and rectangular-MinRank analyses targeting MAYO/QR-UOV [7]; Round-2 parameters reflect updated margins [4].
  • Status: NIST Additional Signatures Round-2 candidate with ongoing cryptanalysis and implementation updates [2], [3].

Implementation notes

  • Arithmetic over \(\mathbb{F}_{16}\); bitsliced/nibblesliced kernels for SIMD vs embedded targets [4], [5].
  • Compact vs expanded keys: public seed expands polynomial blocks on the fly (e.g., via AES-CTR/SHAKE); store only small oil-block data [4].
  • Constant-time sampling and linear-algebra routines; handle the (rare) restart path safely [4].
  • Reference and optimised implementations exist for current Round-2 sets (e.g., liboqs and author repos) [9], [10].

Pros & cons

Pros

  • Very small public keys (≈1–5 KB) and short signatures for some sets (≈186 B) [4], [9].
  • Fast sign/verify; flexible size-vs-speed via \(o,k\) tuning [1], [4].
  • Diversity vs lattice/code-based families in a PQ migration plan [3].

Cons

  • Relies on newer/less-traditional assumptions (OV small-oil + whipped map) [1].
  • Active cryptanalysis space (underdetermined MQ, MinRank variants) → parameters may evolve [7], [8].
  • Careful constant-time linear algebra required for robust side-channel posture [4].
