Origins & history

Classic McEliece is a code-based KEM using binary Goppa codes and a Niederreiter-style construction. It progressed through NIST’s PQC process as a Round-4 candidate. In March 2025, NIST selected HQC for standardization; Classic McEliece was not selected, though NIST noted strong use-cases where caching a large public key and benefiting from small ciphertexts and fast decapsulation is practical. [1][2][3]

Design overview

The KEM is built from the Niederreiter cryptosystem instantiated with binary Goppa codes. A public parity-check matrix \(H\) (derived from a hidden structured code) defines a syndrome map. Encapsulation samples a small-weight error vector \( \mathbf{e} \) and outputs its syndrome as the ciphertext; decapsulation uses a fast algebraic decoder to recover \( \mathbf{e} \). A Fujisaki–Okamoto-style transform wraps the CPA core to achieve IND-CCA2 security. [4]

Mathematical setting

Work over \( \mathbb{F}_2^n \) with a binary Goppa code \( \mathcal{C} \subset \mathbb{F}_2^n \) of length \(n\), dimension \(k\), and designed error-correction capacity \(t\). The secret key encodes a support set and a Goppa polynomial \(g(x)\) of degree \(m\) defining \( \mathcal{C} \); the public key is a permuted parity-check matrix \(H\) that hides this structure. [4]

Public/secret relation

\( \text{sk}=(\text{support}, g(x), \text{permutation}),\qquad \text{pk}=H\in\mathbb{F}_2^{(n-k)\times n}. \)

Syndrome & decoding

Ciphertexts are syndromes \( \mathbf{s}=H\mathbf{e}^T \) for weight-\(t\) errors \( \mathbf{e} \). Decapsulation applies an algebraic decoder exploiting the secret Goppa structure to recover \( \mathbf{e} \) if the weight is within capacity. [4]

Algorithms

Key generation

Generate a binary Goppa polynomial \(g(x)\) over \( \mathbb{F}_{2^m} \), a support set of size \(n\), and a secret permutation. Derive a public parity-check matrix \(H\) hiding the structure. Optionally precompute decoding tables for speed. [4]

Encapsulation (CPA core + transform)

Sample a weight-\(t\) error vector \( \mathbf{e} \) and compute the ciphertext syndrome \( \mathbf{s}=H\mathbf{e}^T \). Derive the KDF input (e.g., hash of \( \mathbf{e} \) and \( \mathbf{s} \)) and output the shared secret. Apply a Fujisaki–Okamoto transform to achieve IND-CCA2 security. [4]

Decapsulation

Decode \( \mathbf{s} \) using the secret Goppa structure to recover \( \mathbf{e} \) with negligible decryption-failure rate. Recompute/verify the transform bindings; output the shared secret or a fallback key if checks fail. [4]

Security at a glance

  • Assumptions: Hardness of decoding random binary Goppa codes at the chosen parameters. [4]
  • Security model: IND-CCA2 via FO-style transform around the Niederreiter core. [4]
  • Status: Not selected for NIST standardization in Round-4 (HQC selected, Mar 2025). [1][2][3]
  • Deployment fit: Favorable when public keys can be cached/transmitted infrequently (e.g., VPNs, file-encryption scenarios) thanks to small ciphertexts and fast decapsulation. [1]

Implementation notes

  • Symmetric: SHAKE/SHA-2 depending on build; KDF per spec. [4]
  • liboqs integration: Multiple Classic McEliece variants are available with standard KEM APIs and OpenSSL provider demos. [5]
  • Performance: Very fast decapsulation; public keys are large (≈1.0–1.35 MB), suggesting streaming/caching strategies on constrained devices. [5][6]
  • Hardening: Constant-time decoding, careful key handling, and validation of ciphertext structure to avoid side-channels. [4][6]

Pros & cons

Pros

  • Small ciphertexts (≈194–208 B) and very fast decapsulation. [5]
  • Mature algebraic decoding with decades of analysis on Goppa codes. [4]
  • Good fit when public keys can be distributed once and reused many times. [1]

Cons

  • Large public keys (≈1.0–1.35 MB) can stress storage/transport if frequently rotated. [5]
  • Not standardized by NIST (as of Mar 2025); fewer turnkey ecosystems than ML-KEM/HQC. [1][2]
